# RAG（Retrieval-Augmented Generation）

## 核心思想

RAG 的做法很直接：在让 LLM 回答问题之前，先从外部知识库中检索相关内容，把检索结果塞进 prompt，让模型"带着参考资料回答"。

解决的问题：
- **幻觉**：模型编造不存在的事实 → 有了检索结果作为依据，生成内容有出处可查
- **时效性**：模型知识截止于训练时间 → 知识库可以持续更新
- **领域知识**：模型没见过的私有数据 → 不需要微调，直接检索

本质上，RAG 是把"让模型记住所有知识"这个不可能的任务，转化为"让模型学会利用给定的参考资料"。

---

## 基本流程

```
离线阶段（Indexing）：
文档 → 分块(Chunking) → Embedding 向量化 → 存入向量数据库

在线阶段（Retrieval + Generation）：
用户提问 → Query Embedding → 向量检索 Top-K 相关片段 → 拼入 Prompt → LLM 生成回答
```

---

## 各环节详解

### 1. 文档分块（Chunking）

将长文档切成适合检索和上下文窗口的片段。分块策略直接影响检索质量：

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| 固定长度 | 按 token 数切（如每 512 token），相邻块有重叠 | 通用，简单粗暴 |
| 语义分块 | 根据段落/章节/语义边界切 | 结构化文档（技术文档、论文） |
| 递归分割 | 先按大分隔符（换行、标题）切，超长的再细分 | LangChain 默认策略 |
| 句子级 | 每个句子一个块，检索后取相邻句子作为上下文 | 精确检索场景 |

重叠（overlap）很重要——如果一个关键信息恰好被切在两个块的边界，没有重叠就会丢失上下文。

### 2. Embedding 与向量数据库

用 embedding 模型把文本块转为稠密向量，存入向量数据库以支持相似度检索。

常用 embedding 模型：OpenAI text-embedding-3、BGE、E5、GTE 等。

常用向量数据库：

| 数据库 | 特点 |
|--------|------|
| FAISS | Meta 开源，纯库，适合嵌入到应用中 |
| Milvus | 分布式，支持大规模生产部署 |
| Chroma | 轻量，适合原型和小规模应用 |
| Pinecone | 托管服务，免运维 |
| pgvector | PostgreSQL 插件，已有 PG 的团队友好 |

### 3. 检索策略

| 策略 | 原理 | 优劣 |
|------|------|------|
| 向量相似度（Dense） | 用 embedding 余弦/点积相似度 | 语义匹配能力强，但可能漏掉关键词精确匹配 |
| 关键词检索（Sparse） | BM25 等传统信息检索 | 精确匹配好，但不理解同义词 |
| 混合检索（Hybrid） | Dense + Sparse 融合排序 | 兼顾语义和精确匹配，目前最佳实践 |
| 重排序（Rerank） | 用 cross-encoder 对初筛结果精排 | 提升精度，但增加延迟 |

### 4. Prompt 组装与生成

将检索到的片段拼入 system/user prompt，常见格式：

```
根据以下参考资料回答用户问题。如果参考资料中没有相关信息，请说明无法回答。

参考资料：
[1] ...
[2] ...
[3] ...

用户问题：...
```

## 进阶优化

| 方向 | 做法 |
|------|------|
| Query 改写 | 对用户原始问题做扩展/分解，提升检索召回率 |
| HyDE | 先让 LLM 生成一个假设性答案，用它做检索（假设答案和真实文档更相似） |
| 多轮检索 | 第一轮检索后根据结果补充检索，迭代细化 |
| 知识图谱增强 | 在向量检索基础上引入结构化知识关系 |
| 自适应检索 | 模型自行判断是否需要检索（不是每个问题都需要外部知识） |

---

## RAG vs 微调 vs 长上下文

| 方案 | 适用场景 | 代价 |
|------|---------|------|
| RAG | 知识库频繁更新、需要溯源、数据量大 | 检索系统的工程复杂度 |
| 微调 | 需要改变模型行为/风格、知识相对固定 | 训练成本、数据标注 |
| 长上下文 | 单次对话内需要参考大量材料 | token 成本高、注意力稀释 |

实践中三者经常组合使用，不是互斥的。

---

## 多模态 RAG

将 RAG 扩展到非文本模态：

- **图文混合检索**：对图片生成描述文本或直接用多模态 embedding（如 CLIP）编码，与文本一起检索
- **表格/图表**：解析为结构化数据或截图后用视觉模型理解
- **音视频**：转录为文本后走标准 RAG 流程，或用专用 embedding

核心挑战：不同模态的 embedding 如何对齐到同一语义空间，以及如何在 prompt 中有效呈现多模态检索结果。

## RAG 的固有缺陷

RAG 不是万能的。以下是通用场景和代码场景下各自的核心问题。

### 通用场景

| 问题 | 说明 |
|------|------|
| **检索召回不全** | 用户问题的表述和文档的表述不一致时，语义检索可能漏掉关键片段。检索不到就等于知识不存在 |
| **噪声干扰** | Top-K 中混入不相关片段，模型可能被误导。检索结果越多，信噪比越难控制 |
| **多跳推理困难** | 答案需要综合多个片段的信息时（A 文档提到 X，B 文档提到 X→Y），单轮检索很难一次性召回所有依赖链 |
| **上下文窗口瓶颈** | 检索片段占用上下文空间，留给实际推理和生成的窗口变小 |
| **误信检索结果** | 模型倾向于相信 prompt 中的内容，如果检索到的文档本身有错或过时，模型会把错误信息当权威引用 |

### 代码场景的特殊困难

代码 RAG 比自然语言 RAG 难得多，原因是代码的语义结构与自然语言有本质差异：

1. **标识符语义对 embedding 模型近乎失效**
   - `getUserById` 和 `fetchUserFromDB` 语义相同但字面完全不同，embedding 未必能对齐
   - 变量名、函数名是高度项目特有的，通用 embedding 模型没有见过这些标识符的"语境"
   - 代码的真正语义往往在调用关系和类型系统中，不在字面文本里

2. **准确率的乘法效应**
   - 代码 RAG 通常是多步流水线：解析代码结构 → 分块 → embedding → 检索 → 排序 → 拼 prompt
   - 如果每步准确率 90%，五步下来整体准确率只剩 $0.9^5 ≈ 59\%$
   - 而代码场景对准确率的容忍度极低——差一个函数签名、少一个类型定义，生成结果就可能编译不过

3. **代码库持续变化**
   - 活跃项目每天都有 commit，索引的时效性以小时计。昨天索引的函数签名今天可能已经改了参数
   - 重建索引的成本不低（需要重新解析、分块、embedding），增量更新逻辑复杂

4. **上下文依赖是图状而非线性**
   - 自然语言文档是线性的，相邻段落大概率相关
   - 代码的依赖关系是图状的：一个函数可能依赖分散在十几个文件中的类型、常量和工具函数，线性分块无法捕捉这种结构

这也是为什么 Coding Agent（如 Claude Code）更多依赖 AST 解析、符号索引、grep 等确定性工具做代码检索，而不是纯粹的向量 RAG。
